Executive Summary

What the investigation found, why it feels like a new issue every day, and what the first two weeks buy

This investigation reconstructed ~35 production incidents (2026-03 → 2026-06-12) from the 96-file memory corpus, 1,224 commits of git archaeology, the live VPS (read-only), and the four health tiers — then adversarially verified a 16-item remediation roadmap against the actual code and the actual box. The headline: the ~35 incidents collapse into 11 cross-cutting failure shapes, and almost nothing since the 2026-05-03 cloud migration is novel. Each "new" incident re-instantiates a pattern that had already fired — usually with a documented lesson on file that was never swept across sibling call sites. The git record confirms the lived experience: 381 of 1,224 commits (31%) are fixes, the fix:feature ratio has risen every month to 1.89 in June, and since 05-12 every single active day contains at least one fix commit.

Four patterns dominate and predict the next incident. (1) Unbounded blocking I/O — and its nastiest subclass, GIL-holding libsql calls on the single FastAPI event loop, which froze the whole API twice on 06-11 and defeated both a thread offload and the deploy gate that should have shipped the fix. (2) Alive-but-dead data planes behind green health — the undiagnosed nightly JVM API-listener wedge fired 78 times in the last 7 days (26 forced gateway restarts, each cascading the relay), and three relay outages delivered zero ticks while every health check passed. (3) A monitoring layer that fails itself — ~10 incidents were caused by the observability stack (false alarms, latched errors, a watchdog that hung 6 hours during the exact outage it exists to catch). (4) Point fixes that never sweep siblings, so each documented lesson re-bites within days on an unaudited twin. Two of these are firing right now: the 06-11 mirror-off hotfix silently inverted the read-freshness contract (scan panels serve Turso snapshots frozen at 06-11 while fresher JSON sits unreachable on disk, and seven services' heartbeats went dark), and a hand-installed beta unit has crash-looped 156,481 times over 10 days with zero alerting.

The "new issue every day" feeling is therefore accurate and structural, not bad luck: incidents are recorded on at least 16 of ~20 trading days since mid-May. The system has roughly 30+ independently-degradable edges (13 units with copy-pasted env, three drifting config sources, five external SaaS dependencies, a serial human-phone 2FA dependency), a gateway recovery machine with a single sensor that goes blind exactly when radon-api is the broken component, and four restart control planes coordinated only by an advisory lock that the prescribed recovery tool (radon restart) bypasses. June's incidents concentrate in the recovery and deploy machinery itself — the first-order app bugs are increasingly handled; the "who watches the watchers" layer is now the dominant failure source.

The roadmap (section 10) sequences 15 verified items into NOW / NEXT / LATER. The six NOW items — all small-to-medium effort — directly disarm the top open exposures in the next-incident ranking (§9.3): they unfreeze the stale scan panels and dark heartbeats, stop the beta crash loop and put flap brakes plus unit-state alerting on every unit, make the 2FA push lock enforced rather than conventional (one restart authority for the gateway), give the deploy pipeline a gate that cannot be wedged by the thing it is deploying, bring all live config under git with a daily drift audit, and flip the embedded-replica default to safe in code. Two focused weeks remove the standing causes of roughly half the recurring incident classes; the NEXT block then attacks the two unresolved roots (the JVM wedge and the sync-libsql class) plus history, metrics, backups, and perimeter CI guards.

Top 5 priorities

  1. DUR-01 — Unfreeze DB-first reads and the silenced heartbeats (firing in production right now: frozen panels + false-stale banner armed for every market morning).
  2. DUR-03 — Enforce the 2FA push lock in the radon CLI and admin panel (the next push-stacking outage is one watchdog race away, and it is the prescribed recovery for everything else).
  3. DUR-02 — Kill the beta crash loop; StartLimit + journald caps + unit-state alerting on all units (156k restarts and a 3.9 GB journal accrued invisibly).
  4. DUR-05 — Make the deploy gate independent of radon-api and add retry to migrate.py (the 06-11 wedge blocked its own fix; a DNS blip hard-failed startup on 06-12).
  5. DUR-07 — Flip the embedded-replica default to OFF in code (one forgotten env var resurrected a 1.36 GB replica and hung the watchdog 6 hours; the unsafe default survives in 2 of 3 clients).

3.4 Single points of failure

  • CRITICAL IB Gateway container is the cascade root and the most failure-prone component. Relay + monitor declare Requires=radon-ib-gateway; a clean gateway stop kills them and Restart=always never re-fires (clean stop is not a crash — see feedback_systemd_cascade_stop_no_autorecover). FastAPI /health can pass while the live data plane (ticks, fills, journal) is dead — exactly the 2026-05-15/05-18 incidents. The Java API listener also wedges while alive (upstream_dead=true), which Docker's restart: always never catches because the process doesn't exit.
  • CRITICAL The 2FA push to one human phone is a hard serial dependency for any gateway restart. Every recovery path (watchdog self-heal, POST /ib/restart, radon restart, deploy-triggered restarts) ultimately funnels through one operator approving one IBKR Mobile push, governed by the cross-process lock at /var/lib/radon/ib-2fa-push-lock.json. Stacked pushes brick all approvals; radon restart still does not hold the lock (feedback_radon_restart_stacks_2fa_with_watchdog).
  • HIGH radon-api is a single uvicorn worker / single event loop. One blocking libsql call on the loop froze every request including /health (commits c9e518a, 2647c93; mirror now gated OFF by default). Every Next.js API route, the admin panel, all autonomous POST timers, and the watchdog's probe go through this one process.
  • HIGH Single Caddy + single box, no swap. Caddy down = total outage including the never-502 floor's transport. 0 B swap on 7.6 GB means an on-box next build during deploy competes with the live stack with no relief valve.
  • HIGH The recovery components themselves are SPOFs: the ib-watchdog oneshot self-hung for 6h (no timeout, then a resurrected replica sync) while it was the only thing that could restart a wedged gateway; the designed autoheal sidecar is absent from the live VPS compose. The system's MTTR depends on watchdog correctness more than on primary-path correctness.
  • MEDIUM Autonomous timers depend on radon-api being up (they POST to :8321). An api wedge silently stops vcg/portfolio/cta/leap refreshes — staleness surfaces only via watchdog windows, not at fire time.
  • MEDIUM Operator CLI /usr/local/bin/radon is the only sanctioned whole-stack control plane, lives in the radon-cloud repo, was installed manually (originally not in setup-vps.sh), and must use polkit — sudo systemctl silently no-ops over non-interactive SSH (radon-cloud 9f29cd2).

Every recovery that cycles the gateway container fires an IBKR Mobile push (unless the SOFT token is still warm).
